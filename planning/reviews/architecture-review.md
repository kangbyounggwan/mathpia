## Overall Assessment

이 계획은 잘 구조화되어 있으며, 서비스 레이어 추상화를 통한 Mock → Supabase 전환 전략이 합리적이다. 그러나 몇 가지 중요한 개선점이 있다.

## 1. 잠재적 위험 (Footguns)

### 1.1 Gemini AI API 키 노출 (Section 7)
**심각도: 높음**
- 현재 `.env`에 `EXPO_PUBLIC_GEMINI_API_KEY`가 있음
- `EXPO_PUBLIC_` 접두사는 클라이언트 번들에 포함됨
- 학습 분석, 힌트, 풀이 도우미까지 추가하면 API 호출이 크게 증가
- **권장**: 프론트엔드만 고도화라도, AI 호출 빈도를 제한하는 로직 추가. 프로덕션에서는 서버 프록시 필수.

### 1.2 AsyncStorage 용량 한계
**심각도: 중간**
- AsyncStorage는 안드로이드에서 기본 6MB 제한
- 문제 50개 + 풀이 이력 + 오답노트 + AI 분석 결과를 모두 저장하면 용량 초과 가능
- **권장**:
  - 데이터 압축 또는 이미지 데이터 제외
  - 오래된 데이터 정리 정책 명시
  - AsyncStorage 대신 expo-file-system 병행 고려

### 1.3 Mock 데이터와 실제 데이터 간 간극
**심각도: 중간**
- Mock 데이터는 항상 일관적이지만 실제 데이터는 그렇지 않음
- 서비스 인터페이스가 Mock 특화되면 Supabase 전환 시 어려움
- **권장**: 서비스 인터페이스에 페이지네이션, 에러 핸들링, 로딩 상태를 처음부터 포함

## 2. 누락된 고려사항

### 2.1 Gemini AI 응답 캐싱 전략 부재 (Section 3.9)
- 동일 학생의 취약점 분석을 매번 API 호출하면 비용과 지연 문제
- **권장**: 분석 결과를 analyticsStore에 캐싱하고, 새 풀이 데이터가 N건 이상 추가될 때만 재분석

### 2.2 Gemini API 응답 형식 안정성 (Section 7)
- AI 응답이 항상 지정된 JSON 형식을 따르지 않을 수 있음
- 기존 geminiService.ts에서도 JSON 파싱 실패 대응이 있지만 불완전
- **권장**:
  - JSON Schema 검증 추가
  - 폴백 값 설정
  - 재시도 로직 (최대 2회)

### 2.3 학부모-자녀 연결 검증
- 학부모가 다른 학생의 데이터에 접근하는 것을 어떻게 방지하는지 명시 안됨
- Mock에서는 문제없지만, 인터페이스에 권한 검증 패턴을 포함해야 향후 전환 시 안전
- **권장**: 서비스 인터페이스에 userId/parentId 파라미터 명시

### 2.4 다국어/접근성 고려 없음
- 앱이 한국어 전용이지만, 접근성(a11y) 고려가 전혀 없음
- **권장**: 최소한 accessibilityLabel 추가 권장

## 3. 성능 이슈

### 3.1 차트 렌더링 (Section 3.10)
- react-native-svg로 레이더 차트, 히트맵을 직접 구현하면 복잡도가 높음
- 데이터 포인트가 많아지면 렌더링 지연 가능
- **권장**:
  - victory-native 사용 검토 (차트 전문 라이브러리)
  - 또는 react-native-chart-kit
  - 커스텀 SVG는 레이더 차트만 사용하고 나머지는 라이브러리 활용

### 3.2 Gemini API 병렬 호출
- 학습 분석 화면에서 취약점 분석 + 문제 추천 + 리포트 생성을 동시에 호출하면 지연
- **권장**: 단계적 로딩 (취약점 먼저 → 추천/리포트는 백그라운드)

### 3.3 문제은행 목록 렌더링
- 50개 이상의 문제를 FlatList로 렌더링할 때 LaTeX 렌더링이 병목
- **권장**:
  - LaTeX는 뷰포트에 보이는 항목만 렌더링
  - 문제 카드에서는 LaTeX 미리보기 대신 텍스트 요약 표시
  - 상세 보기 시에만 LaTeX 렌더링

## 5. 모호한 요구사항

### 5.1 "충분한 Mock 데이터" (Section 6)
- "문제 50개+"가 충분한지 불명확
- 학년별 8문제면 각 단원별 1-2문제밖에 안됨
- **권장**: 최소 100문제 이상, 주요 단원별 5문제 이상

### 5.2 AI 분석 트리거 시점
- 언제 AI 분석을 실행하는지 명확하지 않음 (페이지 진입 시? 수동 버튼? 주기적?)
- **권장**:
  - 학생: "내 학습 분석" 화면 진입 시 캐시 확인 → 없으면 생성
  - 선생님: "AI 진단 실행" 수동 버튼
  - 학부모: 자녀 리포트 화면 진입 시 캐시 확인

### 5.3 오답노트 "3회 연속 정답" 기준
- 연속 정답의 기준이 모호 (같은 날? 다른 날? 시간 간격 필요?)
- **권장**: 최소 24시간 간격의 3회 연속 정답을 "숙련"으로 정의

## 6. 추가 제안

### 6.1 에러 바운더리
- AI 서비스 실패 시 앱이 크래시하지 않도록 React Error Boundary 추가
- 각 AI 호출에 fallback UI 제공

### 6.2 로딩 UX
- Gemini API 호출은 3-10초 소요될 수 있음
- 스켈레톤 UI + 진행 메시지 ("AI가 분석 중입니다...") 필수

### 6.3 오프라인 상태 감지
- Gemini API 호출 전 네트워크 상태 확인
- 오프라인이면 캐시된 데이터만 표시하고 안내 메시지

### 6.4 선생님 Tab에 학생분석 진입
- students.tsx에서 학생 카드 클릭 → student-analytics.tsx로 이동하는 것이 자연스러운 UX
- 별도 탭보다 학생 목록 내 진입점이 더 좋음
